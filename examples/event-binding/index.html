Event Binding

Sub-views and collections can have their events automatically bubbled up to the parent view referenced by their alias. Parent views can also use the events hash to bind to events fired by the parent itself.

View Events

View events can be referenced in the view events hash (convenient if a parent class is triggering the event) by providing the view event name as the events hash key.

View Model

If the model is set on the view using the model constructor property or by using the setModel method of the view, the model events can be bound using the model prefix within view events.

View Collections

Any collection set on the view using addCollection with an alias value will have the collection events available in the view events hash using the alias prefix.

Sub Views

Any sub-view set on the view using addView with an alias value will have the view events available in the view events hash using the alias prefix.

Event Hierarchy

The ':' character in the event key can also be referenced as a parent object within the events hash. So, the following event bindings are equal: 
events: {
	foo: {
		event1: 'onEvent1',
		event2: 'onEvent2'
	}
}
and 
events: {
	'foo:event1': 'onEvent1',
	'foo:event2: 'onEvent2'
}

Below is an example of view, sub-view, model, collection and DOM event bindings 
...
events: {
	'viewEvent': 'onViewEvent', // bind to the 'viewEvent' event triggered from the same view that the events hash is registered
	'myView:someEvent',	// bind to the 'someEvent' event triggered from a sub-view added using the 'myView' alias
	'myCollection:reset', // bind to the 'reset' event triggered from a collection added using the 'myCollection' alias
	'model:all', // bind to all events triggered from the model set on the view
	'click button' // bind to the click event on any 'button' element found within this view's root element
}
